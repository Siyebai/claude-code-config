# Craft — 代码质量 + 审查 + 模式

## 不可变 (CRITICAL)
永不修改原对象，始终返回新副本。`modify(obj, f, v)` ❌ → `update(obj, f, v)` ✅

## 文件组织
多小文件 > 少大文件。200-400行典型，800上限。按功能域组织。

## 错误处理
显式处理每层错误，UI层给友好消息，服务端记录详情。永不静默吞错误。

## 输入校验
系统边界做校验：schema验证 + 快速失败 + 明确消息。不信任外部数据。

## 代码审查
写/改代码后、提交前、安全变更、架构变更、PR合并前强制审查。审查清单：可读性·函数<50行·文件<800行·嵌套<4层·错误处理·无硬编码密钥·测试覆盖率≥80%。

## 严重等级
CRITICAL(安全/数据丢失→BLOCK) | HIGH(Bug→WARN) | MEDIUM(可维护→INFO) | LOW(风格→NOTE)

## 置信度
≥0.8 直接执行 | 0.5-0.8 标注不确定 | <0.5 先研究

## 模型选型
Haiku(90% Sonnet/3x省钱): 轻量agent/代码生成/worker。Sonnet: 主力开发/复杂任务。Opus: 架构决策/深度推理。

## 上下文管理
避免最后20%窗口做大重构/多文件/复杂调试。低敏感：单文件编辑/工具/文档/简单修复。

## 构建故障
build-error-resolver agent → 分析错误 → 增量修复 → 验证。

## 骨架项目
搜实战骨架 → 并行评估(安全+扩展+相关度) → 克隆 → 迭代。

## 设计模式
**Repository**: 数据访问封装在接口后(findAll/findById/create/update/delete)。
**API响应**: `{success, data, error, meta}` 统一信封。

## 代码质量清单
- [ ] 可读命名 ✓ 函数<50行 ✓ 文件<800行 ✓ 嵌套<4层
- [ ] 错误处理完整 ✓ 无硬编码 ✓ 不可变模式
